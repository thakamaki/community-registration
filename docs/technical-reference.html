Project Overview

Purpose

The Community Registration Client (CRC) is a browser-based intake form designed for use at disaster collection points and emergency shelters. Staff (or self-registering evacuees via QR code) complete the form, and on submission a Pulsara Patient Channel is created in real time — including photos — directly linked to the active incident.

Strategic Goals

Beyond operational intake, the CRC represents a broader platform vision:

  • Ultimate goal: Enable flexible, credentialless interactions — allowing people outside of Pulsara to add patients, persons, pets, and belongings to the Pulsara network without requiring a Pulsara account. The QR-token model is the first step toward this.

Core Capabilities

📋 Registration

  • Person, Pet, or Belonging registration types
  • Demographics, address, narrative
  • Associate Person ID for Pet/Belonging linkage
  • Auto-appended field preview in real time

📷 Media

  • Live camera capture (front/rear flip)
  • File upload (drag & drop or browse)
  • Multiple images per registration
  • Uploaded to Pulsara as attachments

🏷️ Wristband Scanning

  • ZXing barcode scanner (no native app needed)
  • Supports CODE-128, CODE-39, CODE-93, CODABAR
  • Wristband ID → Pulsara Identifier field
  • Associate Person scan for Pet/Belonging

📱 QR / Admin

  • Admin tab: login, environment select, incident setup
  • QR code generation with embedded auth token
  • Download / print / copy QR sheet
  • Activity log for real-time debugging

🌐 Multilingual

  • 10 languages: EN, ES, ZH, TL, VI, AR, FR, KO, RU, HT
  • Full UI translation including placeholders & errors
  • RTL support for Arabic

🔒 Security

  • OAuth 2.0 Bearer token authentication
  • All API calls server-proxied via Vercel (no CORS)
  • QR token treated as session credential
  • No data stored in browser (no localStorage)

vercel.json — Proxy Rewrite Rule

All browser requests to /api/pulsara/… are forwarded server-side by Vercel to Pulsara's API. The browser never makes a cross-origin request, eliminating CORS entirely.

{
  "rewrites": [
    {
      "source": "/api/pulsara/:path*",
      "destination": "https://us-internal-demo.pulsara.com/:path*"
    }
  ]
}
⚠️ Switch the destination to https://api.pulsara.com/:path* for production deployment.

Chief Complaint Assembly (at submit)

The chief_complaint field is assembled dynamically. The final string is truncated to 255 characters.

Standard Registration:

[{RegType}] {NarrativeText} [Ph: {phone} | {address} | AssocPersonID: {id}]

Example — Pet registration:

[Pet] Black lab, brown collar [Ph: 555-123-4567 | 100 Oak St, Denver, CO, 80201 | AssocPersonID: 98234]

[Missing Person] {details} [Ph: {phone} | {last known address} | LastSeen: {location} | Desc: {physical} | Reporter: {name} ({relation}) | ReporterPh: {phone}]

[Missing Person] Male, approx 70, diabetic, last seen Tuesday morning [Ph: 555-9876 | 45 Elm St, Houston, TX | LastSeen: Near Minute Maid Park | Desc: Grey hair, blue jacket, walks with cane | Reporter: Maria Gonzalez (Daughter) | ReporterPh: 555-1234]

Field Logic & Conditional Behavior

Missing Person Mode Logic

  • Triggered by ?mode=missing in the URL, set by the admin when generating the QR.
  • Wristband ID section is hidden — not relevant for community-submitted missing person reports.
  • Registration Type radio buttons are hidden; type is hardcoded to Missing Person in the chief complaint.
  • Demographics section relabeled to describe the missing person, not the submitter.
  • Address section relabeled "Last Known Address" with explanatory sub-label.
  • Two additional fields appear in Demographics: Last Known Location and Physical Description.
  • A purple-themed Reporter Information card appears before submit — all fields optional to avoid discouraging submissions.
  • Photo section is promoted with a strong-recommendation callout (photo of the missing person).
  • Header, banner, and submit button all reflect purple Missing Person branding.
  • Print sheet message changes to "Scan to report a missing loved one" and notes public distribution intent.
  • Admin checklist callout appears when Missing Person mode is selected, requiring a dedicated Pulsara mission before QR generation.

Admin Form Mode Selector

  • The Admin tab exposes a two-button "Form Mode" selector: Registration (default, teal) and Missing Person (purple).
  • Selecting Missing Person reveals the pre-requisite checklist callout.
  • The selected mode is appended to the QR URL as &mode=missing.
  • The "Push to Registration Form" button also applies the selected mode to the current browser session.

DOB ↔ Age Mutual Exclusivity

  • When a Date of Birth is entered, age is calculated automatically and the Age field is disabled + dimmed.
  • DOB is preferred — it maps to Pulsara's dob object. Age only maps to age_in_units when DOB is absent.
  • Clearing the DOB re-enables the Age field.

Registration Type → Conditional Fields

  • Person (default): Associate Person ID field is hidden and cleared.
  • Pet or Belonging: Associate Person ID field slides into view directly below the radio buttons. Its value is appended to chief_complaint on submit.
  • Switching back to Person hides and clears the field.

Barcode Scanner Routing

  • Two scan entry points share one camera modal: Wristband ID (Section 2) and Associate Person ID (Section 3).
  • An assocScanMode boolean flag determines which field receives the decoded value.
  • The flag is set when the Associate scan button is pressed and cleared only inside applyScannedValue() after routing — not on cancel — to prevent race conditions with the async ZXing decode callback.
  • Cancelling the scanner resets the flag explicitly via the Cancel button's onclick.

Narrative Preview (Live)

  • A read-only preview panel appears at the bottom of the Narrative card showing exactly which fields will be auto-appended to chief_complaint.
  • Updates in real time via oninput on: phone, address, city, state, zip, reg type radio, and assoc person ID.
  • Rows shown: 🏷️ Registration Type, 📞 Phone, 📍 Address, 🔗 Associate Person ID (only when applicable).
  • Panel hides automatically when all appended fields are empty.

QR Code / URL Token

  • The QR code encodes a URL with two query params: ?incident={id}&ref={base64(id_token)}.
  • On page load, if ?ref is present in the URL, the token is extracted and the Admin tab is hidden (field staff see only the registration form).
  • If ?cp is present, the Collection Point label is appended to the info banner.
  • The QR token is treated as a session credential — the UI warns it expires on logout.

Image Upload Logic

  • Images accumulate in the allImgs[] JS array during the session.
  • First image is uploaded with suppress_alerts=false (triggers one Pulsara notification).
  • All subsequent images use suppress_alerts=true to avoid notification flooding.
  • Each upload failure is logged and skipped — the submission does not abort on image failure.

Language / RTL

  • Language selector applies translations to all visible UI strings immediately via applyLanguage().
  • Arabic (ar) triggers direction: rtl on the registration tab container.
  • Validation error strings are also translated and pulled from the T[lang] object.

Security Considerations

ConcernCurrent ApproachRecommendation
API Authentication OAuth 2.0 password grant → Bearer token stored in JS memory variable (id_token) Token is session-scoped; no localStorage. Acceptable for prototype. For production, consider short-lived tokens or server-side session.
QR Token Exposure Bearer token is base64-encoded in the QR URL (?ref=…) QR sheet should be treated as a password. Tokens expire on logout. Rotate QRs per shift.
CORS / API Proxy Vercel server-side rewrite — browser never sees cross-origin response Vercel proxy does not add authentication — the Bearer token is passed through as-is. Keep Vercel project private or add Vercel access controls.
PHI in Transit All calls over HTTPS (Vercel enforces TLS; Pulsara enforces TLS) Ensure Vercel project is HTTPS-only (default). Do not use HTTP URLs.
PHI at Rest (Browser) No data persisted — no localStorage, no cookies, no IndexedDB Already correct. Data lives only in DOM inputs and JS variables for the duration of the session.
Credential Handling Username/password entered in Admin tab, used once for token fetch, not stored For shared devices, recommend logging out after each admin session. Consider auto-logout on inactivity.

Known Limitations & Configuration TODOs

ItemDetailAction Required
Wristband Identifier Type ID WRISTBAND_IDENTIFIER_TYPE_ID is null in the current code. Wristband post is skipped with a warning log. Obtain the PATIENT_WRISTBAND_NUMBER identifier type ID from your Pulsara administrator and set the constant in index.html.
Chief Complaint 255-char limit All auto-appended fields (phone, address, type, assocID) share the 255-char limit with the narrative text. Long addresses + long narratives may truncate. Consider a character counter that accounts for appended text. Future: use Pulsara's contacts field for phone if available.
Vercel destination URL Currently points to us-internal-demo.pulsara.com. Production requires a separate vercel.json or environment-based rewrite. Update vercel.json destination before going live. Consider Vercel environment variables to switch per environment.
Single-tab Admin access The Admin tab is hidden when the page is loaded via a QR ?ref= URL. Field staff cannot re-login if the token expires mid-shift. Implement token expiry detection and redirect to a login page, or include a hidden "admin unlock" gesture.
No offline support The app requires an active internet connection for all submissions. No Service Worker or local queue. For disaster use cases with unreliable connectivity, consider a local queue (IndexedDB) with background sync.
Image file size No client-side image compression. Large photos from high-res cameras may be slow to upload or rejected by Pulsara. Add canvas-based JPEG compression before upload (target < 1 MB per image).

Automated Testing

Overview

The Community Registration project uses a two-layer automated test suite that runs on every push to the main branch via GitHub Actions. Tests complete in under 90 seconds and immediately flag any broken behavior with a ✅ or ❌ next to each commit.

🧪 Layer 2 — Unit Tests (Vitest)

Tests pure JavaScript logic with no browser and no network. Runs in ~2 seconds. These test the math and rules of the app — field assembly, validation, visibility logic.

File: tests/unit/utils.test.js

🌐 Layer 1 — UI Tests (Playwright)

Tests a real browser loading the actual index.html. Runs in ~35 seconds. These test what a staff member or registrant actually sees and experiences — buttons, fields, form behavior.

File: tests/e2e/ui.spec.js

How to Add a New Test — Guide for Non-Developers

When should I add a test? Any time you add a new field, change a business rule, or fix a bug — add a test that proves it works. Think of each test as a "permanent memory" that the app never forgets that rule.

Adding a Unit Test (for logic/rules)

Unit tests live in tests/unit/utils.test.js. Each test follows the same simple pattern:

it('describes what should happen in plain English', () => {
  const result = functionToTest({ inputs });  // call the function
  expect(result).toBe('expected output');     // assert what you expect
});

Example — adding a test for a new field appended to the chief complaint:

it('appends shelter type when provided', () => {
  const result = buildChiefComplaint({
    narrative: 'Needs assistance',
    shelterType: 'Red Cross Shelter A',
  });
  expect(result).toContain('Shelter: Red Cross Shelter A');
});

Adding a UI Test (for browser behavior)

UI tests live in tests/e2e/ui.spec.js. Each test opens a real browser, interacts with the form, and checks what it sees:

test('new field appears when condition is met', async ({ page }) => {
  await page.goto('/');                              // open the app
  await page.check('input[value="SomeType"]');       // click something
  await expect(page.locator('#newFieldId'))
    .toBeVisible();                                  // assert what you see
});

Future Enhancement Ideas

🔄 Offline Queue

Cache registrations in IndexedDB when offline; background-sync to Pulsara when connectivity is restored. Critical for disaster scenarios with intermittent connectivity.

🔍 Patient Lookup

Add a "Find Existing" mode that searches Pulsara by wristband scan or name before creating a new channel — prevents duplicate records.

🖨️ Wristband Print

After successful registration, generate a printable wristband label (via window.print CSS) showing name, patient ID, and a barcode.

📊 Dashboard View

A read-only dashboard tab showing recent registrations from the current incident (using GET /api/v1/patient-channels with collaboration filter).

🔐 Scoped QR Tokens

Generate short-lived, scoped tokens server-side (e.g. via a Vercel serverless function) rather than embedding the full Bearer token in the QR URL.

🗜️ Image Compression

Client-side JPEG compression via Canvas API before upload. Target ~800 KB per image to ensure fast uploads in low-bandwidth disaster environments.

🌐 Additional Languages

Expand beyond current 10 languages. High-priority additions for US disaster contexts: Somali (SO), Amharic (AM), Hmong (HMN), Portuguese (PT).

♿ Accessibility (a11y)

Add ARIA labels, focus management, and keyboard navigation to fully support screen readers and assistive technology for staff with disabilities.

🪪 Driver's License Scan (PDF417)

Add PDF417 2D barcode decoding to auto-populate First Name, Last Name, DOB, and Address directly from a scanned driver's license. ZXing supports PDF417; requires adding the format to the hint map and updating the field-routing logic.

🔤 Barcode Regex Validation

Allow admins to define a simple regex pattern on the Admin tab (e.g. ^[A-Z]\d{9}$ to require one letter before digits, or .{1,10} to limit length). The scanner would reject non-matching values with a clear error before applying them to the field.

🎤 Voice-to-Text Data Entry

Add a microphone button to the Narrative field using the Web Speech API (SpeechRecognition). Supports multilingual dictation matching the selected UI language — especially valuable for high-volume, fast-paced registration environments.

🧠 AI-Assisted Smart Entry

A single free-text or voice input box (inspired by Pulsara Intelligence) that uses an LLM to parse natural language in any language and auto-populate structured form fields — name, DOB, address, narrative, and registration type. Dramatically reduces time-to-submit for multilingual, non-technical registrants.

🔍 Missing Person Form Mode

A configurable Admin toggle that switches the form into a "Missing Person Report" mode — allowing a community member to submit details about a loved one they cannot locate. Fields shift to describe the missing person (last known location, physical description, contact info for the reporter). Submitted records are tagged and cross-referenced against registered persons in the incident. The mode toggle lives in the Admin tab as a per-session property.
